# Cortex Cloud AWS Organization Connector - Terraform

Terraform implementation of the Palo Alto Networks Cortex Cloud organization-level AWS connector. This module provisions all the IAM roles, policies, and supporting resources needed to connect an entire AWS Organization to Cortex XDR for cloud security posture management.

## Origin

This Terraform project was created based on the CloudFormation template `connectors_aws_cf-**************.yml`, which is the YAML template generated by the Cortex XDR console when onboarding an AWS Organization. The CF template bundles everything into a single monolithic stack with an embedded StackSet and an inline Lambda. This Terraform version decomposes it into modular, auditable `.tf` files while preserving the exact IAM policies, trust relationships, and registration flow expected by the Cortex backend.

## Architecture Overview

The connector operates on two levels:

1. **Management Account** (where Terraform runs) - Creates IAM roles, policies, an SQS queue (for CloudTrail), a Lambda function for registration, and a CloudFormation StackSet.
2. **Member Accounts** (via StackSet) - The StackSet automatically deploys a mirror of the IAM roles and policies into every account within the target Organizational Unit, so Cortex can assume roles across the entire organization.

```
┌─────────────────────────────────────────────────────────────┐

│                                                             │

│  │  (iam.tf)    │  │  (iam.tf)    │  │  (iam.tf)        │  │
│  └──────┬───────┘  └──────┬───────┘  └────────┬─────────┘  │
│         │                 │                    │             │
│  ┌──────┴─────────────────┴───┐   ┌───────────┴──────────┐ │
│  │   Custom IAM Policies      │   │  SQS ← SNS ← CT     │ │
│  │   (Discovery, ADS, DSPM,   │   │     (sqs.tf)         │ │
│  │    Automation)              │   └──────────────────────┘ │
│  └────────────────────────────┘                             │
│                                                             │
│  ┌────────────────────────────────────────────────────────┐ │
│  │          Lambda Registration Function                  │ │

│  └────────────────────────────────────────────────────────┘ │
│                                                             │
│  ┌────────────────────────────────────────────────────────┐ │

│  │  Deploys roles + policies to every member account      │ │

│    • CortexPlatformRole + managed policies                   │
```

## File Structure

| File | Purpose |
|---|---|
| `main.tf` | Terraform/provider configuration and data sources |
| `variables.tf` | All input variables with validation and cross-variable checks |
| `locals.tf` | Computed values: resource names, module conditionals, template version map |
| `iam.tf` | IAM roles and policies for the **management account** (Platform, Scanner, CloudTrail reader) |
| `sqs.tf` | SQS queue + SNS subscription for CloudTrail log notifications |
| `lambda.tf` | Lambda execution role, function, VPC resources, and the registration invocation |
| `stackset.tf` | CloudFormation StackSet definition and instance deployment to member accounts |
| `outputs.tf` | Exported ARNs, queue URL, StackSet ID, and registration result |
| `lambda_source/index.py` | Python Lambda handler that registers the connector with the Cortex backend |
| `generate_tfvars.py` | Helper script to auto-generate `terraform.tfvars` from a Cortex CF template |
| `terraform.tfvars.example` | Annotated example variables file |

## Deployment Order

Terraform handles dependency ordering automatically, but understanding the logical sequence helps during troubleshooting. Below is the order in which resources are created and why each depends on the previous step.

### Phase 1: IAM Policies (iam.tf)

Custom managed policies are created first because the roles reference them.

| Resource | Condition | Purpose |
|---|---|---|
| `aws_iam_policy.cortex_discovery` | Always | Grants read access to ~80 AWS services not covered by the standard `ReadOnlyAccess` policy (API Gateway, GuardDuty, SageMaker, etc.). Required for Cortex asset discovery. |
| `aws_iam_policy.cortex_ads` | `enable_modules.ads` | Grants EC2 snapshot/image management and KMS permissions so Cortex can perform agentless disk scanning (create, copy, share, and delete snapshots). |
| `aws_iam_policy.cortex_dspm` | `enable_modules.dspm` | Grants RDS snapshot, S3, Redshift, DynamoDB, and KMS access for Data Security Posture Management scanning. |
| `aws_iam_policy.cortex_automation` | `enable_modules.automation` | Grants write permissions across EC2, S3, RDS, IAM, KMS, EKS, Lambda, etc., enabling Cortex to execute automated remediation actions. |

### Phase 2: IAM Roles (iam.tf)

Roles are created after policies so they can be attached immediately.

| Resource | Condition | Purpose |
|---|---|---|
| `aws_iam_role.cortex_platform` | Always | The primary cross-account role assumed by Cortex via `sts:AssumeRole`. Trusts the Cortex outpost role ARN with an external ID. Also trusted by `export.rds.amazonaws.com` for DSPM RDS export tasks. Attaches `ReadOnlyAccess`, `SecurityAudit`, `AmazonSQSReadOnlyAccess`, `AWSOrganizationsReadOnlyAccess`, `AmazonMemoryDBReadOnlyAccess`, plus the custom policies above. |
| `aws_iam_role.cortex_scanner` | DSPM, Registry, or Serverless enabled | A second cross-account role used by Cortex scanner services (DSPM scanner, registry scanner, serverless scanner). Attaches `ReadOnlyAccess` + `AmazonMemoryDBReadOnlyAccess` and inline policies for ECR, Lambda, and DSPM artifact access depending on which scanner modules are enabled. |
| `aws_iam_role.cloudtrail_reader` | `audit_logs` enabled + bucket/SNS provided | Federated role assumed via Google Web Identity (`accounts.google.com`). Grants the Cortex CloudTrail collector read access to the S3 bucket and SQS queue containing CloudTrail logs. |

### Phase 3: SQS Queue & SNS Subscription (sqs.tf)

| Resource | Condition | Purpose |
|---|---|---|
| `aws_sqs_queue.cloudtrail_logs` | `audit_logs` enabled + bucket/SNS provided | Receives CloudTrail log delivery notifications from the organization trail's SNS topic. |
| `aws_sqs_queue_policy.cloudtrail_logs` | Same | Allows the SNS topic to send messages to the queue. |
| `aws_sns_topic_subscription.cloudtrail_to_sqs` | Same | Subscribes the SQS queue to the CloudTrail SNS topic. |

### Phase 4: Lambda Registration Function (lambda.tf)

| Resource | Condition | Purpose |
|---|---|---|
| `aws_iam_role.lambda_execution` | Always | Execution role for the registration Lambda. Attaches `AWSLambdaBasicExecutionRole` (CloudWatch Logs) and `AWSOrganizationsReadOnlyAccess` (to read the org ID). Optionally attaches `AWSLambdaVPCAccessExecutionRole` when deploying in a VPC. |
| `aws_security_group.lambda_vpc_sg` | `deploy_in_vpc` + `vpc_id` provided | Managed security group with HTTPS (443) egress only. Created when you supply a VPC ID so the Lambda can reach the Cortex registration endpoint. |
| `aws_lambda_function.connector_registration` | Always | Python 3.12 Lambda that PUTs the connector registration payload to a signed Google Cloud Storage URL. The payload includes the role ARNs, policy ARNs, template versions, and enabled module configuration. |
| `aws_lambda_invocation.register_connector` | Always | Invokes the Lambda exactly once during `terraform apply`. The invocation payload contains all outputs from previous phases (role ARNs, SQS URL, policy ARNs). The Lambda retries up to 5 times with a 3-second backoff. A `postcondition` block ensures the registration succeeded. Uses `lifecycle { ignore_changes = [input] }` so subsequent applies do not re-register. |

> **Why `provisioning_method = "CF"`?** The Cortex backend expects `CF` regardless of the actual IaC tool used. Do not change this to `TF`.

### Phase 5: CloudFormation StackSet (stackset.tf)

| Resource | Condition | Purpose |
|---|---|---|
| `aws_cloudformation_stack_set.cortex_member_roles` | Always | Defines a service-managed StackSet containing a CloudFormation template that mirrors the management account's IAM roles and policies. The embedded CF template is dynamically generated from the same variables and module toggles, ensuring member accounts get identical permissions. Uses `CAPABILITY_NAMED_IAM` and 100% failure tolerance so one account failure doesn't block the rest. |
| `aws_cloudformation_stack_set_instance.cortex_member_instances` | Always | Targets the specified Organizational Unit (or the org root if `organizational_unit_id` is empty). Auto-deployment is enabled so new accounts joining the OU automatically receive the stack. Supports account-level filtering via `include_account_ids` or `exclude_account_ids`. |

## Module Toggles

The `enable_modules` variable controls which Cortex capabilities are deployed. BASE and DISCOVERY are always active.

| Module | Default | What It Enables |
|---|---|---|
| `audit_logs` | `true` | CloudTrail ingestion (SQS queue, SNS subscription, CloudTrail reader role). Also requires `cloudtrail_logs_bucket` and `cloudtrail_sns_arn`. |
| `ads` | `false` | Agentless Disk Scanning - EC2 snapshot lifecycle + KMS grants. Requires `kms_account_ads` and `copy_snapshot_suffix`. |
| `dspm` | `false` | Data Security Posture Management - RDS/Redshift snapshots, S3, DynamoDB scanning. Requires `kms_account_dspm`. |
| `registry` | `false` | Container Registry Scanning - ECR image pull permissions via scanner role. |
| `serverless` | `false` | Serverless Scanning - Lambda function and layer download permissions via scanner role. |
| `automation` | `false` | Automated Remediation - Write permissions across EC2, S3, RDS, IAM, KMS, EKS, etc. |

When `dspm`, `registry`, or `serverless` is enabled, the Scanner Role (`CortexPlatformScannerRole`) is automatically created with the OUTPOST_SCANNER template version included in the registration payload.

## Quick Start

### Prerequisites

- Terraform >= 1.5
- AWS CLI configured with credentials for the **root (management) account** of your AWS Organization
- Trusted access enabled for CloudFormation StackSets in AWS Organizations (see Step 2)
- Python 3 with PyYAML (`pip install pyyaml`) for the tfvars generator script

### Step 1: Download the CloudFormation template from Cortex Cloud

1. Log in to the **Cortex Cloud** console.
2. Navigate to **Settings > Data Sources & Integrations > Add new**.
3. Select **AWS** and **Organization** as the account type.
4. Choose the modules you want to enable (Discovery, ADS, DSPM, etc.).
5. Download the generated CloudFormation YAML template.

> **The `upload_output_url` in the template expires after ~7 days.** If your downloaded template is older than 7 days, go back to the Cortex console and generate a fresh one before proceeding. An expired URL will cause the Lambda registration to fail.

### Step 2: Enable CloudFormation StackSets trusted access

This is a one-time prerequisite on the management account. Run from the management account:

```bash
aws organizations enable-aws-service-access \
  --service-principal member.org.stacksets.cloudformation.amazonaws.com
```

You can verify it was enabled:

```bash
aws organizations list-aws-service-access-for-organization \
  --query 'EnabledServicePrincipals[?ServicePrincipal==`member.org.stacksets.cloudformation.amazonaws.com`]'
```

### Step 3: Generate terraform.tfvars

Run the generator script from the root of this project. This must be executed in the context of the **root (management) account** of the organization:

```bash
pip install pyyaml
python generate_tfvars.py <your-cf-template.yml> -o terraform.tfvars
```

The script extracts the outpost ARN, external ID, template ID, upload URL, module versions, and enabled modules from the CF YAML. It also performs a version compatibility check against the known template versions this Terraform module was built for.

### Step 4: Review and customize terraform.tfvars

The generated file covers the core parameters, but several settings **must be added manually** because they are environment-specific and not present in the CloudFormation template:

**Account filtering** (if limiting scope):
- `include_account_ids` -- deploy only to specific accounts
- `exclude_account_ids` -- deploy to all accounts except specific ones

Account scope must also be configured in the **Cortex Cloud** console when generating the template. Region-based scope filtering is not supported by this Terraform connector.

**CloudTrail integration** (if using an existing trail for audit logs):
- `cloudtrail_logs_bucket` -- S3 bucket name of your organization-wide CloudTrail trail
- `cloudtrail_sns_arn` -- SNS topic ARN for CloudTrail log delivery notifications
- `cloudtrail_kms_arn` -- (optional) KMS key ARN if the trail is encrypted

**VPC deployment** (if the Lambda must run inside a VPC):
- `deploy_in_vpc = true`
- `vpc_subnet_ids` -- private subnet IDs with NAT Gateway routes
- `vpc_id` and/or `vpc_security_group_ids`

See `terraform.tfvars.example` for full documentation of every variable.

### Step 5: Deploy

Run from the management account:

```bash
terraform init
terraform plan -var-file=terraform.tfvars
terraform apply -var-file=terraform.tfvars
```

### Step 6: Verify

After apply completes, check:
- `registration_result` output confirms `Success = true`
- The connector appears in the Cortex Cloud console under **Settings > Cloud Security > Cloud Accounts**
- The StackSet instances are deploying to member accounts (check the CloudFormation console under StackSets)

## Account Filtering (Optional)

By default the StackSet deploys to all accounts in the target OU. You can restrict this using one of two variables (do not set both):

```hcl
# Deploy ONLY to specific accounts:
include_account_ids = ["111111111111", "222222222222", "333333333333"]

# OR deploy to all accounts EXCEPT specific ones:
exclude_account_ids = ["999999999999", "888888888888"]
```

When neither variable is set, all accounts in the OU receive the deployment.

**Important:**
- Account scope must also be configured on the **Cortex Cloud** side when generating the connector template. The Terraform variables control the StackSet deployment, but Cortex needs to know the scope as well.
- These variables must be added **manually** to `terraform.tfvars` -- the `generate_tfvars.py` script does not populate them.
- **Region-based scope filtering is not supported** by this Terraform connector. The StackSet deploys to the current region only.

## VPC Deployment (Optional)

By default the registration Lambda runs in the standard AWS-managed network. To deploy it inside your VPC (required by some enterprise security policies):

```hcl
deploy_in_vpc      = true
vpc_subnet_ids     = ["subnet-private-1", "subnet-private-2"]
vpc_id             = "vpc-0abc123"     # creates a managed SG with HTTPS egress
# OR
vpc_security_group_ids = ["sg-existing"]  # bring your own SG
```

**Requirements:**
- Subnets must be **private** with a route to a NAT Gateway (Lambda ENIs do not receive public IPs)
- The SG must allow outbound HTTPS (port 443) to reach `googleapis.com`

## Outputs

| Output | Description |
|---|---|
| `cortex_platform_role_arn` | ARN of the Platform Role in the management account |
| `cortex_scanner_role_arn` | ARN of the Scanner Role (null if no scanner modules enabled) |
| `cloudtrail_reader_role_arn` | ARN of the CloudTrail reader role (null if audit_logs disabled) |
| `sqs_queue_url` | SQS queue URL for CloudTrail notifications (null if audit_logs disabled) |
| `registration_result` | Full JSON response from the Cortex registration API |
| `stack_set_id` | CloudFormation StackSet ID for member account deployments |

## Key Design Decisions

1. **StackSet uses embedded CF, not Terraform** - AWS CloudFormation StackSets with `SERVICE_MANAGED` permission model require a CF template body. There is no Terraform-native way to deploy IAM roles across all org member accounts via service-managed StackSets, so the StackSet resource dynamically generates its CF template from the same Terraform variables.

2. **Lambda registration instead of CF Custom Resource** - The original CF template uses a `Custom::PublishRoleDetail` CloudFormation custom resource backed by an inline Lambda. The Terraform version uses `aws_lambda_invocation` with `lifecycle { ignore_changes = [input] }` to achieve the same one-time registration without depending on the CloudFormation custom resource framework (`cfnresponse`).

3. **`provisioning_method = "CF"`** - Despite being deployed via Terraform, the registration payload keeps `provisioning_method` as `"CF"` because the Cortex backend validates this field and only accepts `CF`.

4. **Conditional resources via `count`** - Optional modules (ADS, DSPM, Automation, Scanner, CloudTrail) use `count` or `for_each` to conditionally create resources, keeping the state clean when modules are disabled.
